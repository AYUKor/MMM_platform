# MMM Forecast & Optimizer — Product Design Specification V1

**Статус:** утвержденная концепция интерфейса  
**Дата:** 15 июля 2026  
**Назначение:** единый источник требований к UX/UI для frontend-разработки и визуального прототипирования  
**Продукт:** внутреннее веб-приложение для прогноза эффекта будущих рекламных кампаний и оптимизации медиаплана  

---

## 1. Цель продукта

Приложение позволяет маркетологу загрузить **одну будущую рекламную кампанию**, проверить корректность медиаплана, рассчитать шесть сценариев распределения бюджета, получить прогноз дополнительного эффекта рекламы и скачать рекомендованный медиаплан.

Главный вопрос продукта:

> Насколько эффективна будущая рекламная кампания и какой медиаплан выбрать?

Приложение не должно выглядеть как ML-инструмент, notebook или backend-выгрузка. Пользователь не обязан понимать PyMC, posterior, adstock, saturation, beta structure, support registry, R-hat, ESS или устройство optimizer.

---

## 2. Что оценивает система

Система показывает **incremental media effect** относительно сценария «кампании нет».

Она не прогнозирует весь оборот бизнеса и не включает полный baseline спроса, промо, цены, ассортимент, операционные эффекты и все внешние факторы.

Основные пользовательские метрики:

1. **Дополнительный оборот** — p10 / p50 / p90.
2. **ROAS по обороту** — p10 / p50 / p90.
3. **Дополнительные заказы на 100 000 пользователей** — p10 / p50 / p90; дополнительный ориентир.
4. **Изменение среднего чека** — p10 / p50 / p90.
5. **Надежность сценария для планирования** — шкала 1–10.

## 3. Целевые пользователи и роли

### 3.1. Маркетолог

Может:

- создать собственный расчет;
- загрузить одну кампанию;
- проверить входной медиаплан;
- запустить расчет всех шести сценариев;
- следить за прогрессом;
- открыть свои результаты;
- скачать полный отчет и рабочий медиаплан;
- открыть справку и паспорт модели.

Не видит техническую очередь, model package internals, системные журналы и управление пользователями.

### 3.2. Аналитик

Дополнительно может:

- просматривать доступные расчеты;
- видеть расширенные сведения о модели;
- просматривать ошибки расчетов и отчетов;
- регистрировать новую версию model package;
- запускать проверки новой версии;
- сравнивать версии модели;
- изучать технические сведения расчета.

### 3.3. Администратор

Дополнительно может:

- управлять пользователями и ролями;
- видеть состояние системы;
- управлять очередью расчетов;
- изменять приоритет ожидающей задачи с обязательным комментарием;
- запрашивать безопасную остановку расчета;
- управлять активной версией модели;
- выполнять безопасный откат;
- видеть системные ошибки;
- включать режим технического обслуживания.

## 5. Основной пользовательский путь

```text
Вход
→ Главная
→ Новый расчет
```

```text
Загрузил кампанию

```

# Часть I. Визуальная система

## 6. Общая визуальная концепция

Стиль: **high-contrast analytical product**.

Основные принципы:

- черно-белая основа;
- кислотно-зеленый основной акцент;
- крупные заголовки;
- много свободного пространства;
- скругленные карточки с четкой границей;
- характерная смещенная тень у главных акцентных элементов;
- медленно движущиеся абстрактные контурные линии на фоне;
- минимум декоративных цветов;
- сложные данные объясняются простыми карточками, графиками и текстом;
- название и логотип X5 Digital не используются.

Интерфейс должен выглядеть современно и технологично, но не как игровой продукт и не как презентация.

## 10. Анимированный фон

Используются **медленные абстрактные контурные линии**. Они могут напоминать топографическую карту, response curves, потоки данных или вероятностные контуры, но не обязаны образовывать конкретный объект.

Правила:

- полный цикл движения: 35–60 секунд;
- очень медленное смещение и изменение формы;
- без резких волн, вспышек и мерцания;
- прозрачность примерно 5–10%;
- пользователь может отключить анимацию;
- системная настройка «уменьшить движение» отключает движение;
- под графиками и таблицами фон однотонный.

Выраженность:

| Страница | Выраженность |
|---|---|
| Вход | высокая |
| Главная | умеренная |
| Новый расчет | только верхняя часть |
| Процесс расчета | умеренная |
| Результаты | минимальная |
| Таблицы и административные страницы | почти отсутствует |

## 11. Карта России

Используется абстрактная точечная карта, а не стандартная административная карта.

### 11.1. Историческая карта модели

- одна точка = гео;
- яркость точки зависит от общего исторического рекламного бюджета;
- шкала от тускло-зеленого до кислотно-зеленого;
- значения визуально сглаживаются, чтобы крупнейшие гео не скрывали остальные;
- по умолчанию отображается относительная интенсивность, а не обязательные абсолютные суммы.

Рекомендуемая шкала:

- `#26351F`
- `#709D42`
- `#AEEB60`
- `#C7FD72`

### 11.2. Карта будущей кампании

- гео кампании подсвечиваются зеленым;
- яркость и небольшой размер точки отражают бюджет;
- оранжевое кольцо = осторожность;
- красное кольцо = критическая проблема;
- при загрузке точки появляются последовательно и делают один мягкий импульс.

### 11.3. Подсказка точки

Показывает:

- гео;
- бюджет;
- каналы;
- период;
- статус поддержки;
- краткое замечание.

## 12. Графики

### 12.1. Бюджет «было / стало»

- исходный план — серый;
- рекомендованный план — кислотно-зеленый;
- сценарий 5 — зеленый контур;
- предупреждение — янтарный;
- блокировка — красный.

### 12.2. Диапазон p10 / p50 / p90

- p50 — яркая точка или толстая линия;
- p10–p90 — полупрозрачный интервал;
- рекомендованный сценарий — кислотно-зеленый;
- S5 — зеленый контур;
- unsafe — янтарный или красный.

### 12.3. Надежность 1–10

Показывается отдельно от денежных показателей.

| Балл | Интерпретация |
|---:|---|
| 9–10 | Высокая надежность |
| 7–8 | Можно использовать |
| 5–6 | Использовать осторожно |
| 1–4 | Требуется ручная проверка |
| N/A | Сценарий не поддерживается |

## 15. Новый расчет: четыре шага

### 15.1. Шаг 1 — Загрузка кампании

Правило:

> Один файл = одна кампания.

Поддерживаемые форматы:

- XLSX;
- CSV.

Шаблон содержит два отдельных формата:

- Daily;
- Interval.

Если `campaign_name` отсутствует, пользователь вводит название вручную.

Если найдено несколько кампаний, загрузка блокируется. Автоматическое разделение запрещено.

После чтения файла показывается только краткое резюме:

- название;
- период;
- бюджет;
- сегмент;
- каналы;
- гео.

### 15.2. Шаг 2 — Проверка кампании

Общие статусы:

1. Кампания готова к расчету.
2. Кампанию можно рассчитать, но есть замечания.
3. Кампанию нужно исправить.

Проверки:

- структура файла;
- бюджет;
- даты;
- одна кампания;
- сегмент;
- каналы;
- гео;
- гео × канал;
- target;
- похожесть трат на историю;
- качество model signal.

Визуализации:

- бюджет по каналам;
- бюджет по гео;
- точечная карта;
- календарь каналов и daily spend.

Каждое замечание содержит:

- где обнаружено;
- что произошло;
- почему это важно;
- что можно сделать;
- блокирует ли расчет.

Некритические замечания не блокируют расчет. Неизвестные сегменты, каналы, гео, отрицательные бюджеты и расхождение бюджета блокируют переход.

### 15.3. Шаг 3 — Сценарии расчета

Все шесть сценариев рассчитываются всегда. Маркетолог не выбирает набор сценариев.

#### S1 — Как загружено

Ничего не меняется.

#### S2 — Равномерно по всем связкам

Общий бюджет поровну распределяется между исходными `гео × канал`.

#### S3 — Гео выровнены внутри каналов

Бюджет канала сохраняется, гео внутри канала выравниваются.

#### S4 — Каналы выровнены внутри гео

Бюджет гео сохраняется, каналы внутри гео выравниваются.

#### S5 — Самый устойчивый план

Стремится уменьшить серьезность замечаний и удержать план в знакомой исторической зоне. Не добавляет новые каналы, гео или связки.

#### S6 — Адаптивный поиск

Перебирает множество вариантов распределения бюджета между исходными связками и оценивает каждый той же MMM-моделью.

Ключевое объяснение:

- S5 — сначала устойчивость;
- S6 — поиск эффективности с обязательной проверкой устойчивости.

Основной показатель выбора: дополнительный оборот. Остальные показатели — дополнительные.

### 15.4. Шаг 4 — Процесс расчета

Пользовательские этапы:

1. Файл подготовлен.
2. Кампания проверена.
3. Рассчитан исходный план.
4. Рассчитаны сценарии 2–4.
5. Построен сценарий 5.
6. Выполняется S6.
7. Сравниваются сценарии.
8. Формируется рекомендация.
9. Готовится отчет.

Во время S6 показываются:

- запланировано вариантов;
- проверено;
- прошли базовую проверку;
- внутри исторической зоны;
- с предупреждениями;
- не удалось оценить.

Не показываются временный лидер, предварительный RTO и меняющаяся рекомендация.

Пользователь может уйти со страницы; расчет продолжается в фоне. После завершения появляется внутреннее уведомление.

#### Блок «MMM за минуту»

- 150 заранее проверенных фактов;
- новый факт каждые 12–15 секунд;
- максимум два предложения;
- источник доступен по нажатию;
- факты не генерируются LLM во время расчета;
- факты хранятся с категорией, источником, датой проверки и статусом.

### 16.2. Вкладка «Обзор»

#### Главный блок решения

Справа — сценарий 5 как безопасная база.

Правило сравнения:

- победил S2/S3/S4/S6 → сравнение с S5;
- победил S5 → сравнение с S1;
- победил S1 → сравнение с S5.

#### KPI-карточки

1. Дополнительный оборот: p10 / p50 / p90.
2. ROAS: p10 / p50 / p90.
3. Дополнительные заказы на 100 000 пользователей: p10 / p50 / p90, с предупреждением о слабой интерпретируемости там, где это применимо.
4. Изменение среднего чека: p10 / p50 / p90.

#### Почему выбран план

Короткое объяснение:

- лучший эффект среди допустимых вариантов;
- отсутствие блокирующих выходов за историю;
- используются только исходные каналы, гео и связки.

#### Что изменилось

- бюджет по каналам;
- бюджет по гео;
- крупнейшее увеличение;
- крупнейшее снижение.

#### Caveats

Максимум четыре коротких пункта:

- incremental effect, а не total turnover;
- ROAS по обороту, а не прибыли;
- orders — дополнительный ориентир;
- статус исторической зоны.

#### Best raw

Показывается только если:

- best raw отличается от best safe;
- best raw выше по эффекту;
- best raw имеет блокирующее предупреждение.

Он не называется рекомендацией.

### 16.3. Вкладка «Сценарии и надежность»

Главный график — горизонтальные интервалы p10–p90 для шести сценариев.

Переключатели:

- дополнительный оборот;
- ROAS;
- заказы на 100 000 пользователей;
- изменение среднего чека.

Надежность показывается отдельно от денег.

#### Предварительная шкала надежности

1. Похожесть трат на историю — до 4 баллов.
2. Качество модельной поддержки — до 3 баллов.
3. Историческое покрытие — до 2 баллов.
4. Ширина прогнозного диапазона — до 1 балла.

Правила:

- блокирующий выход за историю → не выше 4/10 и запрет auto-recommendation;
- unsupported → N/A;
- diagnostic как основной objective → auto-recommendation запрещена;
- S5 стремится к высокому баллу, но не получает 10 автоматически.

Сводная таблица содержит:

- сценарий;
- p50;
- диапазон;
- ROAS;
- надежность;
- решение;
- возможность автоматической рекомендации.

S6 показывает best safe как основной результат. Best raw отображается внутри деталей S6.

### 16.4. Вкладка «Медиаплан»

База сравнения всегда S1.

По умолчанию показывается рекомендованный сценарий. S5 можно добавить третьим слоем по переключателю.

Верхние карточки:

- общий бюджет;
- перераспределено;
- изменено связок;
- новые связки = 0.

Визуализации:

- бюджет по каналам «было / стало»;
- точечная карта;
- top-10 гео по изменению;
- heatmap `гео × канал`;
- календарь каналов и daily spend;
- детальная таблица.

Heatmap:

- заливка = направление изменения бюджета;
- рамка = качество / предупреждение.

Optimizer не добавляет новые каналы, гео или связки.

Если auto-recommendation отсутствует, raw-план доступен только как «вариант для ручной проверки».

# Часть VII. Acceptance criteria для дизайна

## 33. Общие критерии

Дизайн считается готовым к frontend-разработке, если:

1. Все основные страницы имеют wireframe и состояния.

15. Маркетолог не видит backend internals.
16. Аналитик и администратор получают отдельные технические экраны.
17. Все графики имеют понятные подписи и таблицы.

# Часть VIII. Handoff для frontend / Codex

## 34. Главная граница реализации

Frontend не рассчитывает MMM-метрики и не принимает решение самостоятельно.

Он получает готовые значения и статусы от backend и только отображает их.

- считать adstock;
- считать saturation;

- скрывать backend warnings.

Для визуальной разработки допускаются только явно помеченные demo/sanitized fixtures.
